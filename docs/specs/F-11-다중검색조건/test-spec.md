| 대상 | 시나리오 | 입력 | 예상 결과 |
|------|----------|------|-----------|
| `check_input` | 다중 유효 날짜 | `dpt_dt="20260315,20260316"` | 예외 없음 |
| `check_input` | 다중 유효 시간 | `dpt_tm="08,10,12"` | 예외 없음 |
| `check_input` | 첫 번째 날짜 무효 | `dpt_dt="2026031,20260316"` | `InvalidDateError("날짜가 잘못 되었습니다. YYYYMMDD 형식으로 입력해주세요.")` |
| `check_input` | 두 번째 날짜 무효 | `dpt_dt="20260315,20260230"` | `InvalidDateError("날짜가 잘못 되었습니다. YYYYMMDD 형식으로 입력해주세요.")` |
| `check_input` | 날짜에 비숫자 포함 | `dpt_dt="20260315,2026-03-16"` | `InvalidDateFormatError("날짜는 숫자로만 이루어져야 합니다.")` |

## 통합 테스트

### go_search() -- 조건 인자 전달

| 대상 | 시나리오 | 입력 | 예상 결과 |
|------|----------|------|-----------|
| `go_search` | 인자 없이 호출 (하위 호환) | `srt.go_search()` | `self.dpt_dt`와 `self.dpt_tm` 사용하여 검색 실행 (기존 동작) |
| `go_search` | 날짜/시간 인자 전달 | `srt.go_search(dpt_dt="20260316", dpt_tm="10")` | 전달된 날짜/시간으로 검색 실행 |
| `go_search` | 날짜만 인자 전달 | `srt.go_search(dpt_dt="20260316")` | 전달된 날짜 + self.dpt_tm으로 검색 실행 |

### check_result() -- 다중 조건 순회

| 대상 | 시나리오 | 입력 | 예상 결과 |
|------|----------|------|-----------|
| `check_result` | 첫 번째 조건에서 예약 성공 | 2개 조건, 조건1에서 "예약하기" 반환 | `go_search` 1회 호출, `_check_result_once` 1회 호출, 예약 성공 반환 |
| `check_result` | 두 번째 조건에서 예약 성공 | 2개 조건, 조건1 매진 + 조건2 "예약하기" | `go_search` 2회 호출, `_check_result_once` 2회 호출, 예약 성공 반환 |
| `check_result` | 모든 조건 매진 후 재순회 | 3개 조건, 1라운드 전부 매진 -> 2라운드 조건1 성공 | `go_search` 4회 호출 (3+1), 대기 1회 발생 |
| `check_result` | 단일 조건 (하위 호환) | 1개 조건, 매진 -> 에러 | 기존 동작과 동일하게 대기 후 재시도 반복, 에러 시 예외 전파 |

### run() -- 전체 프로세스

| 대상 | 시나리오 | 입력 | 예상 결과 |
|------|----------|------|-----------|
| `run` | 다중 조건 예약 성공 | 2개 조건 | `login()` 1회, `check_result()` 호출, 성공 시 `_booked_condition` 설정, 알림 발송 |

## 경계 조건 / 에러 케이스

### _normalize_to_list 경계 조건
- `_normalize_to_list("")` -> 빈 리스트 가능성. `__init__`에서 빈 리스트면 ValueError 발생 필요: `"날짜를 1개 이상 입력해주세요."`
- `_normalize_to_list(",,,")` -> 빈 문자열 필터링 후 빈 리스트 -> 동일하게 ValueError: `"날짜를 1개 이상 입력해주세요."`

### 다중 조건 검증 에러
- 날짜 10개 x 시간 6개 = 60개 조건: 조건 수 제한은 없으나, 로그에 전체 조건 수 출력하여 사용자가 인지하도록 함
- 날짜와 시간 중 하나만 다중값: 정상 동작 (카테시안 곱)

### 세션/네트워크 에러 발생 시
- 특정 조건 검색 중 세션 만료: 재로그인 후 **다음 조건으로 continue** (현재 조건 건너뜀)
- 특정 조건 검색 중 네트워크 오류: `NetworkErrorRecovery.recover()`가 현재 조건에 대해 최대 3회 재시도 후 실패하면 `RecoveryError` 전파

## E2E 시나리오

### E2E-01: 다중 조건 중 두 번째 조건에서 예약 성공

**전제 조건**: SRT 인스턴스가 2개 조건 `[{20260315, 08}, {20260315, 10}]`으로 초기화

1. `run()` 호출
2. 로그인 성공 확인
3. 조건 1 (`20260315, 08`) 검색 실행 -> 매진 (예약 가능 좌석 없음)
4. 로그 출력 확인: `"검색 조건: 날짜=20260315, 시간=08"`
5. 조건 2 (`20260315, 10`) 검색 실행 -> "예약하기" 발견
6. 로그 출력 확인: `"검색 조건: 날짜=20260315, 시간=10"`
7. `book_ticket()` 호출 -> 예약 성공
8. `is_booked == True` 확인
9. `_booked_condition == {"dpt_dt": "20260315", "dpt_tm": "10"}` 확인
10. Telegram 알림 발송 확인 (dept_time: "10")

### E2E-02: 모든 조건 1라운드 매진 후 2라운드에서 예약 성공

**전제 조건**: SRT 인스턴스가 3개 조건으로 초기화

1. `run()` 호출
2. 로그인 성공
3. 1라운드: 조건 1, 2, 3 모두 매진
4. 로그 출력 확인: `"모든 조건 확인 완료. {N}초 대기 후 다시 처음부터 검색..."`
5. 대기 후 2라운드 시작
6. 2라운드 조건 1에서 예약 성공
7. `is_booked == True` 확인
8. `cnt_refresh == 1` 확인 (1라운드 완료 시 1 증가)

### E2E-03: 단일 조건 입력 시 기존 동작 유지 (회귀)

**전제 조건**: SRT 인스턴스가 단일 조건 `{20260315, 08}`으로 초기화

1. `run()` 호출
2. 로그인 성공
3. 조건 검색 -> 매진
4. 대기 후 동일 조건 재검색 반복
5. 3회차에서 예약 성공
6. `is_booked == True` 확인
7. 기존 `check_result()` 동작과 동일한 흐름 확인

### E2E-04: 검색 중 세션 만료 발생 시 복구 후 계속 검색

**전제 조건**: SRT 인스턴스가 2개 조건으로 초기화

1. `run()` 호출
2. 로그인 성공
3. 조건 1 검색 실행 -> 세션 만료 감지 (URL이 로그인 페이지로 리다이렉트)
4. 자동 재로그인 실행
5. 재로그인 성공 후 **조건 2로 건너뛰어** 검색 계속
6. 조건 2에서 예약 성공
7. `is_booked == True` 확인

## 회귀 테스트

| 기존 기능 | 영향 여부 | 검증 방법 |
|-----------|-----------|-----------|
| 단일 `--dt`, `--tm` 입력 | 영향 있음 (내부 표현이 list로 변경) | `SRT("동탄", "동대구", "20240115", "08")` 기존 테스트 전체 통과 확인 |
| `check_input()` 검증 | 영향 있음 (루프 구조 변경) | 기존 `TestSRTInputValidation` 12개 테스트 전체 통과 확인 |
| `go_search()` 호출 | 영향 있음 (인자 추가) | 기존 인자 없이 호출 시 self 속성 사용 (하위 호환) |
| `check_result()` 루프 | 영향 있음 (루프 구조 변경) | 기존 `TestSRTCheckResult` 2개 테스트 수정 필요 (go_search 호출이 check_result 내부로 이동) |
| `book_ticket()`, `reserve_ticket()` | 영향 없음 | 기존 테스트 그대로 통과 |
| `run()` 메서드 | 영향 있음 (go_search 호출 제거) | 기존 `run()` 흐름 테스트 수정 필요 |
| `refresh_result()` | 영향 없음 | 기존 테스트 그대로 통과 |
| F-08 에러 리커버리 | 간접 영향 (check_result 내부 호출 위치 변경) | 네트워크/세션 복구 동작 확인 |
| F-09 Telegram 알림 | 간접 영향 (notify_success 호출 인자 변경 가능) | 알림 메시지에 조건 정보 포함 확인 |
